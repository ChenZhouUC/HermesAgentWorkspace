# Schema as Handoff Contract (DDL 作为生产者/消费者契约)

当一条数据管线被切分为多个独立服务时，"两段之间如何握手"是架构核心。传统答案是 RPC、消息队列、共享缓存。**DDL 作为契约**是另一类答案——上下游通过共同维护的数据库 schema 而非 API 握手；schema 即 API，权责由代码而非数据库权限隔离。

## 模式定义

两个服务（生产者 P、消费者 C）约定：

1. 在同一个 schema registry 里**同时声明** P 产出的表与 C 产出的表，让双方对全部数据形态有 single source of truth；
2. 但**代码层强制单写权责**——P 只写自己的表，C 只写自己的表，反过来读对方的表；
3. 这种"双方都拥有 DDL 知识、各自只能写一半"的设计让 schema 演化成为双方共同关心的变更点。

[[trajex]] 与 [[hidalgo]] 计算层之间正是这种关系：trajex 声明上游特征表与下游结果表的全部 DDL，但只对前者写入；Hidalgo 反向消费特征表并写入结果表。两边都对 schema 演化有完整视野，无需通过 protobuf / OpenAPI 等额外定义层维持接口。^[[[_living/Whale-SpaceSight/ReID-Perception-Layer-trajex|ReID-Perception-Layer-trajex]]]

## 与传统握手方式的对比

| 握手方式             | 接口定义位置                        | 演化成本                     | 数据保留               |
| -------------------- | ----------------------------------- | ---------------------------- | ---------------------- |
| **RPC（gRPC/REST）** | proto/OpenAPI 文件                  | 改接口需 P、C 同步发版       | 不保留（请求响应即时） |
| **消息队列**         | 消息 schema（Avro/Schema Registry） | 兼容性规则受限于序列化框架   | 短期保留               |
| **DDL 契约**         | 数据库 schema 定义                  | 列级演化（ALTER + 双写过渡） | 长期保留可查询         |
| **共享缓存**         | 散落在双方代码中                    | 高耦合，演化困难             | 短期保留               |

DDL 契约的特殊优势：

- **历史数据可重新消费**：消费者升级后可以反向重跑老数据，不像 RPC 那样"响应一旦返回就丢失"；
- **schema 演化天然有缓冲期**：列可以"先加上、双写过渡、清理旧列"，比 RPC 的版本号管理直观；
- **观测性好**：所有握手数据都是查询可达的表行，便于 debug、统计、回溯；
- **不需要额外的服务发现 / 鉴权 / 限流基础设施**——数据库本身就承担这些。

## 适用与不适用

**适用**：

- 离线/准实时分析管线（小时/天级延迟可接受）；
- 数据量大、需要长期保留可查询（如审计/合规场景）；
- 上下游迭代节奏差异大（一方快、一方慢，RPC 反复对齐版本成本高）；
- 同一组织内部两个服务（schema 共享需要组织信任）。

**不适用**：

- 在线低延迟链路（毫秒级响应）；
- 跨组织 / 跨公司握手（schema 不能作为外部 API 暴露）；
- 数据量小、形态变化频繁的握手（数据库不是好的高频小消息中转）。

## 工程实现要点

- **schema 文件版本化**：DDL 进 git，两边代码仓库通过 submodule 或 publish package 引用同一份 schema 定义；
- **代码权责隔离而非数据库权限**：保留"双方代码上都有完整 schema 知识"的设计意图，便于工具链复用与本地测试——单纯靠数据库 GRANT 区分写权限会丢失这层语义；
- **列级演化模式**：加列免迁移，改列用"双写过渡 + 切换 + 清理"三步，删列必须先停所有消费方再执行；
- **分区策略与契约同管理**：按天/按租户分区是 P/C 双方共同关心的——分区策略变化会同时影响两边的查询模式；
- **观测性钩子**：每行数据带生产时间、数据来源、模型版本等血缘字段，让消费方能区分"这批数据来自哪个生产者版本、哪个回放任务"。

## 与影子部署的协同

[[model-shadow-deployment|配对特征轴影子部署]]本质上是这套模式的一个特例：生产者用"双特征轴写入"在 schema 内部演化模型版本，消费者通过修改"读哪一组字段"完成 cutover，无需 schema 大版本变更。两个模式叠加后，整条管线的版本演化几乎可以做到"全部在配置层完成"。
