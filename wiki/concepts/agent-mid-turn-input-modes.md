# Agent Mid-Turn Input Modes (插入式输入调度)

当用户在 Agent 一个回合（Turn）尚未结束、还在工具调用或思考链路中时追加新消息，Runtime 需要决定如何调度这条消息。三种基础模式构成对偶：**打断 (interrupt) / 排队 (queue) / 引导合并 (steer)**——每种模式在"是否终止当前回合"和"是否保留 Prompt Cache"这两个轴上有明确取舍。该机制由 [[hermes-agent]] 在 macOS Gateway 中实现并通过热切换命令暴露。^[[[_living/AI-Applications/Hermes-Agent-macOS-Ops|Hermes-Agent-macOS-Ops]]]

## 三模式对偶

| 模式        | 是否终止当前回合     | 是否保留 Prompt Cache          | 是否消耗新回合数       | 典型用途               |
| ----------- | -------------------- | ------------------------------ | ---------------------- | ---------------------- |
| `interrupt` | 是（杀死并重启）     | 否                             | 是                     | 立即叫停跑错方向的任务 |
| `queue`     | 否（串行等候）       | 是（旧回合）/ 否（新回合首轮） | 是（每条消息独立回合） | 一口气下达多条独立任务 |
| `steer`     | 否（合并到当前回合） | 是                             | 否                     | 补充约束、补正前置指令 |

## 打断模式 (interrupt)

默认模式。若 AI 正在执行耗时工具调用（运行代码、抓网页、长推理），Runtime 立刻丢弃并终止当前回合，再以最新到达的消息作为新的用户输入开启新回合。**牺牲 Prompt Cache 换取响应性。**

### 空闲降级幻觉

如果用户极快连发两句，第一句尚未触发耗时工具（Runtime 仍在 Idle），由于没有正在跑的任务可打断，Runtime 自动**降级为 queue 行为**——两句被分别瞬时回复。这容易给用户造成"底层是并行 / 按顺序分别处理"的错觉，实际上是 interrupt 没有命中靶子之后的兜底路径。

## 队列模式 (queue)

在首个回合彻底结束前，所有新到达的消息进入内存队列。第一个任务生成最终回复后，依次取出队头消息开启下一轮**全新回合**。**串行而非并行**——两条消息不会同时占用同一份对话上下文。

## 引导合并模式 (steer)

插队式上下文合并。新消息不打断当前任务，也不开新回合，而是作为"引导注释"无缝注入到**下一个工具调用结束后的系统上下文**——模型在同一回合的后续思考中即时看到这条补充。

- **Prompt Cache 友好**：前缀未变，KV Cache 命中率不受影响；
- **不消耗新回合数**：在按回合计费 / 限额的运维语境下尤其划算；
- **触发方式**：配置 `/busy steer` 后默认走 steer，或在任意模式下强制前缀 `/steer 补充内容`。

## 选择启发式

- 想撤掉当前方向 → `interrupt`
- 想批量提交独立任务 → `queue`
- 想给当前任务补一条约束 → `steer`

`interrupt` 与 `steer` 的核心差异不是"快慢"，而是**是否保留正在跑的那份上下文**——保留就用 steer，必须丢弃就用 interrupt。
