# ReID Pipeline (行人重识别系统流水线)

线下零售门店行人重识别 (ReID) 系统的端到端架构方法学。具体实现中，[[hidalgo]] 是该项目的顶层名称，也常指负责后续聚合、角色推断与结果回写的计算层服务；[[trajex]] 是同一项目中的特征推理服务。ReID 的输出仅是行人级原子事实（去重后的行人 ID + 角色 + 轨迹），业务指标（客流量、接待）由独立的下游后处理层负责，详见 [[customer-flow-post-processing]]。

## 采集 / 计算 / 编排：三段分层

[[hidalgo]] 计算层核心是一个**离线批处理函数**：`(店铺, 起止时间, 模式) → ReID 结果`。它不采集、不推理、不调度，三类职责分别由上下游承担：

- **上游（感知层）**：[[trajex]] 服务把图像→检测→特征向量；上装类别、角色得分、行人特征预先算好后写入特征表；底库相似度查询（通用角色服饰库 / 公司或店铺工服库 / 店员库）也在这一层完成（见 [[reid-library-lookup]]）；
- **核心（Hidalgo 计算层）**：从特征表读已计算好的 Embedding 与轨迹，做角色推断 / 重判 + 多层级聚合 + 结果回写；
- **下游（编排层）**：DAG 调度器按店/按时间窗触发核心；本系统只暴露一张轻量任务进度表给调度层做幂等与续跑。

这种"采集 / 计算 / 编排"切分让模型升级不动批处理；调度策略变化不动算法；算法改阈值不动调度——是后续所有可演进性的基础。^[[[_living/Whale-SpaceSight/ReID-Pipeline-Architecture|ReID-Pipeline-Architecture]]]

## 三模式对偶：Entrance / Interior / Coupling

核心运行有三种模式，每次调用只跑一种：

- **Entrance**：出入口轨迹去重——把同一顾客多次出入口轨迹聚为同一行人；
- **Interior**：店内轨迹组装——把店内多摄像头碎片轨迹聚为完整店内动线；
- **Coupling**（生产主模式）：先用 Entrance 模式产出"锚轨迹"，再把店内轨迹匹配挂载到对应行人 ID 上，形成"全店动线"。

Coupling 是 **Entrance 复用 + Interior 匹配** 的组合，不是独立算法路径——避免了"全店"这条复杂主线被特化逻辑污染。

## 角色判定的两阶段：预判 + 后重判

角色判定**不是单一前置步骤**，而是穿插在聚类前后：

### 预聚类（per-image / per-trajectory）

- **逐图打分**：根据上装检测几何信号（头盔→外卖/快递、徽章→店员）+ 上装相似度（工服库 / 特殊角色库）逐图给出角色类别；
- **逐轨融合**：把全轨迹的图级别打分加权聚合到轨迹级；店员判定用**店内特征模型分** 与 **上装统一模型分** 的加权融合，避免单一信号在光照/角度上的不稳定；
- **预筛**：聚类前用 `ped_type` 白名单筛掉与该模式无关的角色。

### 聚类后（per-cluster）

- **重判**：聚类完成后，对每个簇取众数角色；
- **徽章比例覆写**：若簇内徽章命中轨迹占比足够高，强制改判为店员；
- **融合分兜底**：若簇内某轨迹的融合分超阈值，再次改判为店员。

这种"前判定 + 后重判"承认了**聚类本身可以修正角色**——一条孤立的店员轨迹会被识为顾客，但当它和真店员的其他轨迹合到同一簇后，整簇会被改判回店员。^[[[_living/Whale-SpaceSight/ReID-Pipeline-Architecture|ReID-Pipeline-Architecture]]]

## 评估指标分层

为了解耦**特征模型** 与 **聚类策略** 与 **下游业务**，评估指标按层：

- **角色标签层**：精确率 / 召回率（按角色类别分别评）；
- **聚类策略层**：双向熵——一边算"过度聚合"，一边算"过度切分"。两边都要小才算稳，详见 [[multi-stage-clustering]]。

更上层的特征模型性能（Rank-N / mAP 等）由**训练侧**独立评估，不在本系统职责内——这是上下游分层职责的体现。

## 基础设施选型要点

### 向量列与业务列同表

老系统用专用向量数据库 + 业务关系库双写，存在一致性与权限治理问题。把向量列直接放进关系库（利用其向量扩展），让向量与业务字段落在同一张表里：

```
Database  ─▶ Schema    ─▶ Table      ─▶ Partition ─▶ Column
功能区        公司          特征/结果     时间天分区     向量 / 业务字段
```

按店扩缩、按天归档、按公司隔权限全部走一套 ACL 与 DDL；向量列与业务列同事务读写，从源头消除"向量库与业务库双写一致性"问题。

### 推理服务的本机优化

[[trajex]] 感知层的模型推理通过推理服务统一暴露；与该服务同机部署时，走**共享内存直连**而不是 HTTP——绕过反序列化与 socket 开销，是把端侧重的推理服务挪到生产环境时的关键加速点。

### 外部调度器 + 本地进度表

生产环境用外部 DAG 调度器周期触发本系统，本系统**只暴露一张轻量任务进度表**记录每次调用的店铺 / 时间窗 / 模式 / 状态 / 耗时——既是任务级幂等的真理源，也让算法与调度互不绑架。

### 按店参数覆写

默认参数之外提供一张**每店参数覆写表**：以 JSON 列存放每个店铺的差异化参数，运行时与默认参数做 deep merge。新策略上线只是改这张表里某些店的配置项，不动主代码、不动镜像。

## 反作弊与采样

按日采样上限内做**有偏下采样**——优先保留"进店"事件而下采样"出店"事件，因为业务侧的关键指标是进店人数。上限本身按店可调。

## 多层级聚合

ReID 单纯靠 Embedding 单阶段聚类性能不够，门店间方差极大；必须采用按"由严到松、由同向到异向、由轨迹到批次"分多层 tier 的策略，详见 [[multi-stage-clustering]]。
