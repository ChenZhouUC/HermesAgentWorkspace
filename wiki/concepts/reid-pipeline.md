# ReID Pipeline (行人重识别系统流水线)

线下零售门店行人重识别 (ReID) 系统的端到端架构方法学：把多路摄像头上传的轨迹切片与抓拍图像处理为去重后的 _行人 ID + 角色标签_。它的输出仅是行人级原子事实，业务指标（客流量、接待）由独立的后处理层负责，详见 [[customer-flow-post-processing]]。

## 前后端解耦的架构原则

ReID 系统只承担三件事：

1. **跨摄像头、跨时段聚合**：把同一真实行人产生的多条轨迹收敛到同一个行人 ID；
2. **角色判定**：每条轨迹打上顾客 / 店员 / 外卖 / 快递 / 保安等标签；
3. **原子结果对外暴露**：以 ReID 结果 + 业务宽表的形式供下游消费。

它**只输出原子级行人事实**，不计算业务指标。这种"前后端解耦"使算法侧能独立迭代特征模型与聚类阈值，下游业务策略可以独立灰度上线，互不绑架。^[[[_living/AI-Applications-and-Ops/ReID-Pipeline-Architecture|ReID-Pipeline-Architecture]]]

## 多模型串行的设计模式

整条管线由若干模型节点串行组成，每个节点回答一个原子问题：

- **服装/身体检测**：定位上装、头部、安全帽等部件 bbox，并基于部件标签直接短路出某些特殊角色（如检测到安全帽即归入外卖/快递/保安）；
- **上装特征比对**：把上装特征与多个底库（工服库 / 特殊角色库）做相似度查询，命中即判定角色——这些命中样本**不进入顾客聚类池**；
- **场景化行人特征**：出入口与店内分别用独立的特征模型，避免单一模型在两种构图、光照差异下被迫折中；
- **质量过滤**：低质量帧打分丢弃，避免污染特征空间；
- **跨场景统一特征**：再用一组通用特征模型为全店级聚合提供共享空间。

通用经验：**让每个模型只回答一个问题**，串成可独立替换、独立评估的节点。任何节点升级都不应改变下游接口语义。

## 角色过滤先于聚类

聚类前必须先剔除无关角色，否则不同身份的轨迹混入同一聚类空间会严重污染顾客池。固定执行顺序：

```
[快递 / 外卖 / 保安] ─▶ 店员 ─▶ 顾客（仅顾客进入聚类）
```

工程经验上，**单一工服模型识别店员不够**：在光照特殊或背景复杂的场景下相似度容易处于混淆区，准确率不稳定。必须**联合 [工服库 + 全店店员底库]** 双重识别，且两类底库都需要多角度、多光照样本——这是把店员召回拉到生产可用水位的关键。^[[[_living/AI-Applications-and-Ops/ReID-Pipeline-Architecture|ReID-Pipeline-Architecture]]]

## 评估指标的三层分层

为避免「特征模型不行」与「聚类策略不行」相互掩盖，评估指标按三层解耦：

- **模型效能与鲁棒性**：Rank-N、mAP、mCP（Complete Precision，完全召回下的精确度）；
- **角色标签效果**：准确率、召回率；
- **聚合策略效果**：双向平均熵分布（GT→PV 避免漏聚合 / PV→GT 避免误聚合）、类别数量比。

这种分层后，特征模型升级与聚类策略迭代可以并行评估，不相互污染。

## 基础设施选型要点

### 向量存储与业务库统一

老系统的常见做法是用专用向量数据库 + 业务关系库双写，存在一致性与权限治理问题。重构思路：把向量列直接放进关系库（利用其向量扩展），利用关系库的原生层级直接映射门店组织架构：

```
Database  ─▶ Schema    ─▶ Table      ─▶ Partition ─▶ Column
Functionality ─▶ Company ─▶ Shop      ─▶ Floor/Region ─▶ Vector
```

这层映射让"按店扩缩、按天归档、按公司隔权限"全部走一套 ACL/DDL；向量列与业务列同表，从源头消除向量库与业务库双写一致性问题。

### 调度与编排

清洗、聚类、特征生成全部走有向无环图（DAG）调度：

- **容器化 Executor + 自动伸缩**：Worker 按任务负载弹性伸缩，应对数据量增长；
- **多镜像 Executor**：不同 Task 用不同基础镜像，解决底层库编译兼容性；
- **分布式计算引擎接入**：大规模数据走分布式算子，避免单机瓶颈；
- **Rollout 策略**：DAG 进版本控制 + 影子部署 + 金丝雀 + 蓝绿环境，平滑升级回滚。

## 聚类策略

ReID 单纯靠 Embedding 单阶段聚类性能不够，门店间方差极大，必须采用分阶段降维分治——该方法学已抽出为独立概念 [[multi-stage-clustering]]，本文不重复展开。
