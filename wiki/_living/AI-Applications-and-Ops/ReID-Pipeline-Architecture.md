# ReID 系统架构与重构方案

本文记录线下门店行人重识别（ReID）系统的端到端架构思路、聚类策略与基础设施选型原则。
ReID 是上传轨迹/图像信息的第一道处理；其输出（行人 ID、行人角色）会被下游的客流后处理进一步加工成业务指标，后处理逻辑见 `Customer-Flow-Post-Processing.md`。

## 1. 系统定位与重构动因

线下零售门店在出入口与店内部署多路摄像头，边缘端持续上传**带时空坐标的人体轨迹切片 + 抓拍图像**。ReID 系统的职责是：

1. 在同一门店内把不同摄像头、不同时间窗的轨迹聚合为同一行人；
2. 识别每条轨迹的角色（顾客 / 店员 / 外卖 / 快递 / 保安）；
3. 把识别结果输出到下游做客流统计、接待识别、人货匹配。

老系统的典型痛点：

- 算法只用 Embedding + 单阶段聚类，全量门店上限受限、门店间方差极大；
- 工程层是单点架构，图片下载与向量计算成为瓶颈；
- 用专用向量数据库单独存储 Embedding，与门店组织架构（公司 / 门店 / 摄像头）的层级映射不自然，且与业务库存在双写一致性问题。

重构方向：聚合质量提升至生产可用水平、改造为分布式架构以支持线下店扩张、资源利用率上调、故障自愈。

## 2. 数据链路

整条链路自上而下：

```
边缘相机 ─▶ 消息中转 ─▶ 模型推理服务（多模型流水线） ─▶ 特征写入向量库
                                                  ─▶ 离线聚类与角色判定（编排调度）
                                                  ─▶ ReID 结果（行人 ID + 角色）
                                                  ─▶ 业务宽表（供下游消费）
```

- **特征分层**：图像特征（部件检测、上装特征、行人特征、质量判定）与轨迹特征（轨迹起止时间、坐标统计、属性）分两类入库；前者按帧、后者按轨迹聚合。
- **采样策略**：店内单条轨迹做等距代表帧采样；轨迹级特征则使用原始密度，不再降采样——保留时序细节。
- **分区策略**：按时间范围做物理分区，按天滚动；按 _轨迹 / 摄像头 / 门店 / 公司_ 维度建索引。分区由调度任务自动滚动创建与归档。

## 3. 模型链路设计原则

每条上行图像顺序通过若干模型节点，每个节点只回答一个原子问题：

1. **服装/身体检测**：定位上装、头部、安全帽等部件 bbox，并基于部件标签直接短路出某些特殊角色（如检测到安全帽即归入外卖/快递/保安）；
2. **上装特征比对**：把上装特征与多个底库（工服库 / 特殊角色库）做相似度查询，命中即判定角色——这些命中样本**不进入顾客聚类池**；
3. **场景化行人特征**：出入口与店内分别用独立的特征模型，避免单一模型在两种构图、光照差异下被迫折中；
4. **质量过滤**：低质量帧打分丢弃，避免污染特征空间；
5. **跨场景统一特征**：再用一组通用特征模型为全店级聚合提供共享空间。

通用经验：**让每个模型只回答一个问题**，串成可独立替换、独立评估的节点。任何节点升级都不应改变下游接口语义。

## 4. 多阶段聚类策略

单纯使用 Embedding 矩阵做单阶段聚类性能与鲁棒性欠佳，门店间方差极大。重构后采用 **降维分治、由严到松、由在线到离线** 的多阶段策略：

1. **Online 高确信度聚合**：高频次跑，Single Linkage + 高阈值，保证高准确率，先完成"最容易做对"的那部分去重；
2. **Offline 轨迹级聚合**：闭店后离线进行，结合"时长策略 + 综合相似度"对极相似 / 极短时差的残余轨迹做配对，使用适中阈值；
3. **属性约束增强**：把"进店轨迹"的人体属性作为硬约束（**性别一般可用，年龄不太可用**），限制综合分阈值后对两组轨迹强制合并；
4. **兜底召回**：宽松阈值扫尾剩余碎片。

**核心经验**：每多一轮聚合，召回率单调上升。前序轮已大幅收敛搜索空间，后序可放宽匹配条件，用较低 precision 换更高总 recall，不会污染全局。

### 角色过滤的执行顺序

聚类之前必须先剔除无关角色，否则会污染顾客池：

```
[快递 / 外卖 / 保安] ─▶ 店员 ─▶ 顾客（聚类对象）
```

店员识别的工程经验：单纯工服模型在光照特殊/背景复杂场景下易处于混淆区，准确率不稳定。必须**联合 [工服库 + 全店店员底库]** 双重识别，且两类底库都需要多角度、多光照样本——这是把店员召回拉到生产可用水位的关键。

### 出入口特殊场景的进出聚合

老逻辑用固定规则判断轨迹是否合并，对店员多次进出这类规律性轨迹判错（误识为多个顾客）。新逻辑基于：

- **轨迹规律性**：进出交替、进出平衡；
- **摄像头一致性**：来源摄像头的分布比例。

动态调整合并条件，使规律性轨迹被正确去重。该过滤项**默认关闭**，按门店在配置中显式开启——因为它对"出入口差异显著"的门店才有效，普适开启反而引入误判。

## 5. 基础设施选型

### 5.1 向量库与业务库统一

老系统用专用向量数据库 + 业务关系库双写，存在一致性与权限治理问题。重构把向量存储迁到关系库的向量扩展上，利用关系库原生层级：

```
Database  ─▶ Schema    ─▶ Table      ─▶ Partition ─▶ Column
Functionality ─▶ Company ─▶ Shop      ─▶ Floor/Region ─▶ Vector
```

这层映射让"按店扩缩、按天归档、按公司隔权限"全部走一套 ACL/DDL；向量列与业务列同表，从源头消除向量库与业务库双写一致性问题。

### 5.2 任务编排

清洗、聚类、特征生成等任务全部走有向无环图调度：

- **K8s 调度器 + 自动伸缩**：Worker 节点按任务负载自动伸缩，应对数据量增长的 Scale 问题；
- **多镜像 Executor**：不同 Task 用不同基础镜像，解决底层库编译兼容性；
- **分布式计算引擎**：大规模数据走分布式计算算子，避免单机瓶颈；
- **Rollout 策略**：DAG 进 Git 版本控制 + 影子部署 + 金丝雀发布 + 蓝绿环境，平滑升级回滚。

### 5.3 标注与评估工具

标注工具用 HTTP/2 多路复用替代 HTTP/1.1 的浏览器并发连接上限，解决海量轨迹图集中标注时的图片加载瓶颈——本质是用协议层并发能力换工具层吞吐。

## 6. 评估指标的三层分层

为了解耦**特征提取效果**与**聚类策略效果**，把指标拆三层：

- **模型效能与鲁棒性**：Rank-N、mAP、mCP（Complete Precision，完全召回下的精确度）；
- **角色标签效果**：准确率、召回率；
- **顾客聚合策略效果**：双向平均熵分布（GT→PV 避免漏聚合 / PV→GT 避免误聚合）、类别数量比。

这套指标让算法侧能独立调试特征模型，不被聚类阈值或角色错分污染。

## 7. Rollout 原则

按"准备 → 小范围 → 扩大 → 全量 → 长期监控"分阶段：

- 小范围阶段观察足够长时间，覆盖不同业务周期；
- 扩大阶段引入更多业务多样性（不同规模、不同业态）；
- 全量分批迁移，旧系统保留一段时间作回滚后备；
- 全程维护对照表，记录新旧系统的对应关系，便于 bad case 复盘。

主要风险点：上游数据 DDL 延迟、数据分布变化导致下游指标飘移、机器配置与代码逻辑的取舍。
