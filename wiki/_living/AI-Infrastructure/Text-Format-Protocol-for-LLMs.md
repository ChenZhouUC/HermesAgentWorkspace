# AI 时代文本载体协议：基于大模型交互场景的格式架构设计

在 AI 时代，文本载体正在发生底层重构。Markdown 逐渐取代一部分富文本和传统二进制文档，成为大模型（LLM）交互中的常用格式。这并不是偶然，而是由 Token 成本、结构可解析性和训练语料分布共同推动的结果。

## 一、为什么大模型时代偏好 Markdown？

1. **Token 效率高**：相比 HTML 或 Word 底层 XML，Markdown 用较少字符（如 `#`、`-`、`*`）表达标题、列表和代码块，通常能降低 Token 消耗，并在上下文窗口中保留更多有效信息。
2. **语义边界清晰**：Markdown 的标题、列表和代码块围栏为模型提供了明确的结构边界，便于区分指令、背景、数据和代码。
3. **训练语料覆盖广**：GitHub、StackOverflow、Reddit 等高质量开源与技术社区中大量使用 Markdown，使其成为许多模型在预训练阶段高频接触的文本形态。

## 二、基于不同受众场景的文本与多媒体架构

针对不同的人机协作阶段，信息封装格式应有所区分。

### 场景一：AI 读，人不读

适用场景：AI to AI 通信、System Prompt、RAG 知识检索、系统日志。

核心诉求：确定性、低 Token 占用、严格解析规则。人类阅读体验不是重点。

推荐文本格式：

- YAML / JSON：用于传输结构化参数或工具调用（Tool Calling）。YAML 相比 JSON 少了部分括号和引号，通常更省 Token，但严格机器解析时仍要注意缩进和类型歧义。
- 极简结构化 Markdown：用于传递背景知识（Context）或系统指令。保留标题、列表、代码块等结构，减少加粗、斜体等纯排版语法。

推荐多媒体格式：

- 图片 / 视频：使用 URL 绝对路径，或在必要时使用 Base64 编码输入多模态模型。
- 图表 / 逻辑关系：转化为纯文本的 Graph 节点列表、JSON 树或矩阵数组。AI 通常更需要底层结构，而不是渲染后的视觉图像。

### 场景二：AI 和人都读

适用场景：Chat 交互上下文、Prompt 模板设计、人机协作文档、技能库。

核心诉求：人机认知同频。既要方便人类阅读和编辑，也要让 AI 能稳定理解逻辑层级。

推荐文本格式：

- GitHub Flavored Markdown (GFM)：在人类可读性和机器可解析性之间取得较好平衡，也便于在飞书、Obsidian、GitHub 等工具间迁移。

推荐多媒体格式：

- 图表 / 架构图：推荐 Mermaid.js 或 PlantUML。人类端可渲染为流程图、时序图，AI 端则可直接读写纯文本定义。
- 数学公式：使用标准 LaTeX 语法块。
- 图片：使用带语义描述的 Markdown 图片语法，让 Alt text 成为 AI 理解图片上下文的提示。

### 场景三：人读，AI 不读

适用场景：最终分析交付物、数据 Dashboard、汇报演示文稿、BI 看板。

核心诉求：降低认知负担、提升可视化表达效率和信息吸收速度。

推荐文本格式：

- 原生 UI 组件 / HTML+CSS：如飞书 Block 卡片、Notion 交互式页面。分栏、卡片、高亮块等视觉结构更适合人类快速浏览，但对原生模型来说解析成本较高。

推荐多媒体格式：

- 交互图表：通过 ECharts、D3.js 等渲染可悬停、下钻的可视化图表。
- 音视频组件：用于最终消费和演示的 MP4 或流媒体组件。
- 状态呈现：使用状态徽标、进度条等视觉元素替代冗长文字说明。

## 三、架构总结

在 AI 软件工程中，数据流转可以按以下层次设计：

1. 底层（AI 通信领域）：优先 YAML / JSON。
2. 中层（人机协作领域）：优先 Markdown / Mermaid。
3. 表层（人类消费领域）：优先卡片化（Cards）和可视化 UI。
