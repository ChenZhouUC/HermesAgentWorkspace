# AI Agent 框架分析与选型指南

本文从工程开发视角梳理主流 AI Agent 框架的定位和适用场景。相关生态变化很快，具体版本能力应以各项目官方文档和最新发行说明为准。

## 1. 核心认知澄清：什么是真正的“Agent 框架”？

在当前的 AI 领域，“Agent”这个词被严重泛化。很多框架标榜自己是 Agent 框架，但它们产出的“东西”完全不同。

从工程开发的视角，一个真正符合直觉的 Agent 框架（Agent Harness/OS）应该具备以下特征：

1. 开发者视角：工程师只需要提供工具（Tools/API/数据库连接）和系统设定（System Prompt）。
2. 框架视角：框架在底层接管大模型的调度循环（思考 -> 调用工具 -> 观察结果 -> 再次思考），处理死循环、上下文截断、工具入参错误重试等复杂逻辑。
3. 产物视角：产出一个能够自主决策、长期运行、解决复杂开放性任务的智能程序实例。

基于这个核心认知，可以将常见框架分为以下五类。

## 2. 框架阵营全景图

### 阵营一：真正的 Agent 底座（Harness / OS）

特点：接管底层 ReAct 循环，是私有化 Agent 的核心引擎。开发者主要负责接入工具和业务上下文。

#### 1. Dify (平台化 / 低代码)

- 定位：企业级 LLM 应用开发与编排平台。
- 产物：一个带有可视化 UI 后台、可以直接生成前端对话框或 RESTful API 的成型软件系统。
- 开发流：90% 时间写后端微服务并生成 YAML，10% 时间在 UI 拖拽编排。
- 特性：私有化部署路径清晰，底层编排和调度能力较成熟。

#### 2. Agno (原 Phidata，纯代码)

- 定位：专为构建“Agent 软件”设计的 Python 极简轻量级框架。
- 产物：一个纯 Python 工程，作为后台服务轻量级部署。
- 开发流：80% 时间写纯 Python 业务函数，20% 时间通过一行代码实例化 Agent。
- 特性：开箱即用体验较好。支持记忆持久化、多模态输入、基于 Pydantic 的函数级工具调用、知识库（RAG）以及监控和前端 UI 能力。

#### 3. Pydantic AI (强类型 / 工程化极致)

- 定位：追求极致类型安全与依赖注入的 Agent 框架。
- 产物：类型约束更强、结构化输出更稳定的 Agent。
- 开发流：70% 时间定义 Pydantic Schema 和全局依赖，30% 时间注入给 Agent。
- 特性：极度克制。专注解决大模型输出格式错乱及企业全局状态安全传递痛点。

#### 4. DeepAgents (LangChain 官方 / 复杂长线任务)

- 定位：专为“长线、重度产出（Long-Horizon, Artifact-Heavy）”设计的固化 Agent Harness。灵感来源于 Claude Code、Manus 和 Deep Research。旨在解决开放式、长链路、需要极长上下文的复杂任务。
- 产物：自带虚拟文件系统、长期记忆和子智能体生成能力的重型 Agent（类似开源版 Devin/Claude Code）。
- 开发流：通过定义大模型、工具、系统提示词进行实例化，复杂的 Context 管理交由框架的文件系统和 Todo 列表完成。
- 特性：把上下文管理从“把历史塞进对话框”转向文件系统、任务列表和规划器（Planner），更适合长周期、多步骤、需要中间产物沉淀的工程任务。

#### 5. OpenAI Agents SDK (OpenAI 官方原生)

- 核心定位：OpenAI 官方轻量级 SDK，重点支持工具调用、Handoff 和多 Agent 协作。
- 适用场景：如果技术栈主要使用 OpenAI 模型，用这个 SDK 开发可以减少适配成本，并获得较稳定的 Tool Calling 与 Handoff 支持。

### 阵营二：多智能体编排与工作流（Multi-Agent / Workflow）

#### 6. LangGraph (LangChain 官方 / 工业级状态机)

- 核心定位：抛弃了让大模型“自由发挥”的黑盒机制，强行把 Agent 的工作流画成了一张“有向图（Graph）”和状态机。
- 适用场景：适合对流程可控性要求高的生产系统，例如需要 Human-in-the-loop、确定性重试、状态持久化或多节点编排的业务。

特点：解决单体 Agent 搞不定复杂业务流的问题，通过拆分角色、分发路由来完成工作。

#### 7. CrewAI (流水线与角色扮演)

- 定位：静态角色扮演与多智能体协作框架。
- 特性：大模型不再自由发挥，而是被限制在既定的 SOP 内流转。

#### 8. AutoGen (微软出品 / 群聊辩论)

- 定位：对话驱动（Conversation-Driven）的多智能体框架。
- 特性：通过多角色对话和丰富扩展钩子实现协作、评审和自我修正，适合代码生成与研究型实验。

#### 9. Swarm (OpenAI 官方轻量级 Handoff)

- 定位：轻量级的网络路由与交接（Handoff）框架。
- 特性：通过 Handoff 函数移交上下文，适合轻量级任务分发和路由实验。

### 阵营三：代码驱动与沙盒执行（CodeAgents）

特点：颠覆传统的 JSON 函数调用，直接让大模型写代码来干活。

#### 10. SmolAgents (Hugging Face 出品)

- 定位：代码即行动（Code as Action）的沙盒执行框架。
- 特性：大模型直接输出 Python 代码并在本地安全沙盒一次性运行，可缓解海量数据通过 JSON 函数调用传输时的 Token 压力。

### 阵营四：数据管道与系统集成（Copilot / RAG）

特点：将大模型与现有的企业私有数据或庞大的微服务架构进行深度缝合。

#### 11. LlamaIndex (数据与检索)

- 定位：数据框架与 RAG 领域的重要工具。
- 特性：通常挂载到 Agno、Dify 或自研服务中，承担数据连接、索引和检索增强生成能力。

#### 12. Semantic Kernel (微软出品)

- 定位：企业级 Copilot 插件化（Plugin Architecture）总线。
- 特性：以业务逻辑为中心，将 ERP 接口封装为原生插件，内部 Planner 自动编排执行图。

#### 13. RAGFlow (深层文档检索与 Agentic RAG)

- 核心定位：不是用来写代码的，而是专门用来啃硬骨头文档的“重型知识库 Agent”。
- 适用场景：当文档中包含复杂表格、扫描件或排版复杂的 PDF 时，RAGFlow 的深层解析能力更有价值，适合做企业知识库和 Agentic RAG。

### 阵营五：Prompt 编译器

特点：抛弃手写提示词，用机器学习的方式自动优化 Agent。

#### 14. DSPy

- 定位：声明式、编译型的大模型框架。
- 特性：通过声明式模块和优化器自动改写提示词或示例，适合离线优化复杂 Prompt 管线，再把优化结果回注到生产代码中。

## 3. 私有化 Agent 二开选型与组合推荐

根据“定义工具 / 数据库 / API，部署到自己服务器，尽量少碰底层调度”的二开诉求，可按场景选择：

- 场景 A：极致开发效率与业务系统集成核心底座：Dify（适用希望开箱即用的 B 端产品化团队）
- 场景 B：极客掌控力与纯 Python 逻辑驱动核心底座：Agno（适用追求轻量化的单体业务首选）
- 场景 C：传统系统集成与强一致性后端重构核心：Semantic Kernel（C#/Java 首选）；格式防弹：Pydantic AI。
- 场景 D：庞大企业中枢与智能分发核心编排：Swarm（利用 Handoff 构建企业总闸，无缝分发背后的专属 Agent）
- 场景 E：重度数据分析与报表生成核心底座：SmolAgents（沙盒直接跑代码，避免海量数据 JSON 传输导致 Token 爆炸）
- 场景 F：极长周期的复杂软件工程核心底座：DeepAgents（自带虚拟文件系统和任务列表引擎，让大模型读写本地沙盒和待办清单，而不是把所有日志塞进模型上下文）。
- 场景 G：多模态感知与垂直业务系统强绑定（如安防、IoT 设备分析）核心底座：Agno（多模态支持较好，可结合 Pydantic 函数级工具挂载内部工单和控制 API）。
