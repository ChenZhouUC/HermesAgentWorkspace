# 边缘端数据采集算法 (Edge Data Collection ALGO)

本文记录 SpaceSight 产品线中运行在边缘计算盒子（如 RK3576, Sophon 平台）上的核心算法工程库。
这是整个 ReID 与客流分析流水线的**最前置节点**。

## 1. 系统定位与职责

在整体架构中，边缘端算法服务位于物理摄像头与云端感知层（TRAJEX）之间：

```
边缘相机 ─▶ 边缘计算盒子（运行本 ALGO） ─▶ 弱网缓存与消息队列 ─▶ 云端 TRAJEX（感知层）
```

**核心职责**：

- **视频流接入与解析**：直接拉取边缘相机的 RTSP/RTP 视频流进行硬解码。
- **目标检测与追踪**：在边缘端执行轻量级的人体/车辆检测与多目标追踪。
- **事件与属性抽取**：
  - 识别并提取出入口事件（Entrance/Exit）与店内事件（Interior）。
  - 截取代表性的轨迹图片。
  - 抽取时空信息：坐标序列、时间戳序列。
  - 提取扩展属性：如年龄、性别等结构化信息。
- **数据上报**：将打包好的轨迹切片和图像数据上传至云端消息队列，供下游 TRAJEX 消费。

## 2. 弱网缓存与离线续传机制

线下门店的网络环境通常不稳定，边缘端设备必须具备高可用性。算法在数据上报模块中内置了**本地缓存机制**：

- **正常网络**：数据实时通过消息队列/API 上传到云端。
- **弱网 / 无网环境**：上传失败的轨迹和图片数据会持久化存入盒子本地的存储（如 SQLite 或本地文件队列）中。
- **网络恢复**：后台常驻的守护线程会检测网络连通性，一旦恢复，自动将本地堆积的离线数据补传到云端，确保云端 TRAJEX 与计算层的召回率不会因短暂断网而受损。

## 3. 代码库与工程结构

算法核心代码位于 `~/Documents/WhaleRepo/whcode/cv/algo`。
该仓库采用 C/C++ 编写，基于 CMake 构建，支持多平台交叉编译（ARM, Hisi, x86）。

**关键配置与场景复用**：
代码复用于**客流（Customer Flow）**和**车行（Vehicle Flow）**两种场景，打包时通过配置文件区分：

- 客流配置文件：`config/service_default_keliu.json`（带有 `cxc` 前缀/后缀的资源通常代表客流/橙小橙）。
- 车行配置文件：`config/service_default.json`（带有 `ayh` 前缀/后缀的资源通常代表车行/爱义行）。

编译与打包通常由专属的 shell 脚本（如 `package_dianneikeliu_hisi.sh`, `package_monitor_default.sh`）完成，输出对应硬件平台的固件包。

## 4. 业务功能侧实现细节

### 4.1 视频与图像抽取策略

边缘端在处理视频流时采用了**质量评估与动态抽帧**策略：

- **人脸质量评估与打码 (Face Quality & Anonymization)**：每一帧在提取时都会进行人脸评分（包含 `blur` 模糊度评分阈值、遮挡判定等）。为符合数据合规与隐私保护，算法内置了脱敏（Anonymization）逻辑，具体调用神经网络模型的 `BlurImpl` 算子对检测到的高清晰度人脸区域进行局部高斯模糊（Gaussian Blur）或像素化处理，确保上传到云端的抓拍图像去隐私化。
- **轨迹图像上传控制**：为节省带宽和存储资源，系统通过 `up_traj_cnt`（轨迹最大上传张数）和 `up_traj_track_len`（轨迹跟踪最小像素跨度）等配置，限制了同一轨迹内打包上传图片的数量。
- **消息发送通道**：所有的事件在打包后，最终会转换成指定格式并通过 Kafka / MQTT 协议投递至云端或本地网关。

### 4.2 事件判断逻辑

算法通过分析轨迹并结合逻辑预设定义了以下典型的业务事件：

- **进出事件 (Entrance / Exit / Door Event)**：依靠设置在画面上的虚拟触发线或区域（AreaMap），结合运动轨迹的交叉判定来辨别顾客的物理进店与出店。同时系统支持通过 HTTP 接口动态向后台（`api.meetwhale.com/shop/analyzer/getentrance`）拉取/更新出入口多边形或检测线配置，若拉取失败将退回本地配置或上报告警。
- **驻留与关注事件 (Approach / Front / Passby)**：门店分析演进为面向整个“门店空间”或“展区”的人货场分析：
  - **路过 (Passby)**：通过计算轨迹向量在店外方向上的平行投影比值（`passby_walk_ratio`）与宽高比（`passby_wh_ratio`），当目标仅在门外边缘移动且未深入 AreaMap 时，判定为纯路过，有效剔除“门框边缘徘徊”的伪进店噪音。
  - **驻留 (Approach)**：根据追踪目标在某一指定区域的持续持留时间戳（`last_hold_timestamp`）是否超过配置阈值，判断其是否在该区域产生停留。
  - **关注 (Front)**：在 Approach 的驻留基础上，如果目标的身体或人脸朝向（基于检测框坐标与角度推算）正对有效展区/关注区内，则判定为发生正脸交互的关注（Front）事件，代表高意向的行为。
- **特殊角色事件 (Worker / Vehicle)**：系统配置了专门的 `WorkerAreaProcessor` 和 `VehicleProcessor`，分别负责店员（基于工作区域/轨迹规律过滤）与车辆轨迹提取，这使得一份基础库能够在客流和车行不同场景下平滑复用。

### 4.3 基础属性提取

除了位移，Edge ALGO 还负责顾客本体生理特征的**极前置推断**（利用端侧轻量化模型）：

- **性别与年龄 (Gender & Age)**：对提取到的质量最高的人脸/人体目标应用属性检测模型（Attribute Model，包含 `attri.age` 与 `attri.gender` 算子），输出预测的性别和预估年龄。
- **口罩与帽子 (Mask & Hat)**：输出是否佩戴口罩的检测结果（`attri.mask`）。这些结构化的小数据附着在事件（Event）的 JSON 体内，与图片流分离，使得云端在进行二次计算或客群画像时极为高效。
