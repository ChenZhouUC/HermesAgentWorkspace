# 客流后处理与指标加工逻辑

本文记录 SpaceSight 系统中，从原始轨迹到最终业务指标的数据加工全生命周期。
客流后处理在架构上是一条严格的“**由端到云、由感知到业务**”的漏斗链路。从边缘计算盒子到离线 Airflow 调度，整个过程分为三个顺序阶段：感知侧基础过滤 ➔ 中台清洗与二次过滤 ➔ 业务指标衍生与加工。

> **代码边界说明**：
>
> - **感知侧**：主要位于门店端计算盒子（Edge ALGO）与核心云端服务（TRAJEX / HIDALGO）。它们负责极前置的物理判断，输出“去重后的行人 ID + 角色 + 轨迹”。
> - **数据中台侧**：位于独立的下游服务（基于 Airflow 调度的离线自动化脚本，代码主仓为 `whcode/data/airflow/analytics`）。负责复杂的业务规则清洗与深度的指标加工。

---

## 第一阶段：感知侧基础过滤 (Edge ALGO & HIDALGO)

在数据刚生成、尚未进入业务统计域之前，系统首先依赖视觉和物理时空几何进行第一层"防脏水"拦截：

1. **过店 / 门框边缘徘徊 (Passby)**：
   - 依赖极前置 **Edge ALGO** 的平行投影比值（`passby_walk_ratio`）与宽高比。
   - 叠加 **HIDALGO** 聚类层基于像素跨度和单调方向的检查。将未深入店内的轨迹直接标记为无效。
2. **静态物体与环境扰动**：
   - **假人 / 样品检测**：服装/汽车门店的假人因被搬运或检测框抖动产生轨迹。通过外包围框极小或异常稳定、以及行进方向无明确矢量来判定并剔除。
   - **嵌套伪人 (特殊发型)**：发型被误识为头部导致的"套娃"双重检测框，通过上衣置信度偏低及相对位置关系剔除。
3. **视觉身份极早期过滤**：
   - 结合 TRAJEX 的 1-NN 工服底库检索与 HIDALGO 聚类簇内表决，直接丢弃极高置信度的“店员”、“安保”等非顾客轨迹。

## 第二阶段：中台二次清洗与过滤 (Airflow Data Cleaning)

数据落库后，由 Airflow 接管（如 `customer_flow/reid_unify/` 系列脚本）。这里引入了强业务属性的规则，进一步修剪不符合门店运营口径的数据：

1. **营业时间与时区过滤 (Time & Timezone Filter)**：
   - 根据每家门店（`shop_id`）配置的具体营业时间（如早 10 点 - 晚 10 点），剔除店员盘点、闭店盘货或非营业时间段内的无效轨迹。
   - 跨国/跨区门店需做精确的时区边界对齐。
2. **逛店停留时长过滤 (Stay Duration Filter)**：
   - 按店开启时长黑名单/白名单（`stay_duration_status_open_list`）。
   - 将轨迹时长（如进店到离店时间差）过短的记录清洗掉（可能是极速穿堂或误识入库）。
3. **高频人员与进店次数过滤 (Frequency / Count Deduplication)**：
   - 基于 ReID 的跨日、全天去重结果，针对"外卖/快递去重"、"跨天店员去重"、"轨迹自定义次数过滤"设定优先级。
   - 对于频繁出现（进店次数异常高，如商场保洁、隔壁店员）的轨迹 ID 进行降权或抹除。
4. **特殊角色后置过滤 (Role Filter)**：
   - 结合业务后台配置的特定逻辑，再次对部分未被 HIDALGO 拦截边缘角色（如儿童、特定服装等）进行过滤，输出高纯净度的顾客底表。

## 第三阶段：核心业务指标加工 (Airflow Analytics)

经过彻底清洗的轨迹表，最终进入 Airflow 的核心聚合算子（如 `batch/`, `shop_stay/`, `roi_traffic/`, `shop_runner.py` 等），衍生出前端页面展示的复杂商业指标：

### 1. 批次客流聚合 (Batch Flow)

- 现实中顾客常以“家庭/情侣”为单位进店。Airflow 脚本（如 `batch_customer_flow.py`）会基于同轨迹时间窗交叠率、同进入时间等，将多个离散的行人 ID 聚合为一个**客流批次 (Batch)**。

### 2. 接待指标体系 (Reception Metrics)

判断销售是否有效服务了顾客，不仅依靠视觉距离，还大量结合 Pad 埋点与轨迹重叠（`pad_batch.py`，`veriflex_task`）：

- **接待人数 / 接待批次 (`count_acc_ped`, `count_acc_batch`)**：顾客轨迹段与店员轨迹/服务事件有效重合的次数。
- **及时接待率 (`count_acc_in_time_batch`)**：计算“进店时间”与“接待开始时间”的时间差，判定是否在规定时间（如 30 秒）内响应。
- **无人接待批次 (`count_no_acc_batch`)**：全程无人交互的冷板凳批次。
- **深度接待 (`acc_effect_batch`)**：交互时长超过深度判定阈值的优质接待。

### 3. 在店时长与区域热度 (Dwelling & ROI Traffic)

- **在店/逛店时长 (Shop Stay / Shop Duration)**：合并同批次内多次捕捉的时间碎段，计算顾客在整个门店生命周期内的总驻留时长。
- **区域停留时长 (ROI Traffic)**：结合门店展区（AreaMap）划分，统计具体车系、货架、沙盘前的停留人数与驻足时间，形成门店热力图漏斗。

### 4. 店外与路过客流 (Outside Traffic)

- `outside_traffic.py` 分析摄像机视野边缘未触发进店条件但被捕捉到的路过客群，结合进店人数计算出门店核心转化指标：**进店率 (Capture Rate)**。

---

## 4. 配置化设计与调度

整套中台加工体系采用高度**配置化与插件化**：

- 几乎所有的过滤阈值、营业时间、是否开启停留过滤，都是维护在 Postgres 表或 JSON 配置中。
- Airflow DAG 按调度周期（如 `advance_hourly_new_reid_dag`, `ten_minute_dag`, `advance_today`）执行运算，将最终清洗和衍生出的业务结果同步至 **Pinot**（用于前端高速 OLAP 查询）和关系型数据库中。
