# ReID 感知层：TRAJEX 服务架构

本文记录 ReID 系统中 **感知层（perception layer）** 的架构方法学：边缘上传的轨迹 + 图像如何被加工为下游 HIDALGO 计算层可消费的特征数据。
本文与 `ReID-Pipeline-Architecture.md` 配对——后者描述 HIDALGO 计算层（聚类、角色推断、ID 分配），本文聚焦上游。

**命名边界**：TRAJEX 是 `project_hidalgo_reid` 项目中的特征推理子项目/服务；HIDALGO 是顶层 ReID 项目名，也常指下游负责聚合与角色推断的计算层服务。二者都是内部项目名，但服务边界不同。

## 1. 在三段分层中的定位

ReID 整体架构按 _采集 / 计算 / 编排_ 三段切分（见 `ReID-Pipeline-Architecture.md`）。TRAJEX 是其中的**采集与计算前置**段：

```
边缘相机 ─▶ 消息中转 ─▶ TRAJEX（感知层：检测+特征+底库查询）─▶ 特征表
                                                            ─▶ HIDALGO 计算层（聚类+ID）
                                                            ─▶ ReID 结果表
```

**关键边界事实**：

- TRAJEX 与下游 HIDALGO 计算层之间**只有数据库表握手，没有 RPC、没有内存共享**；
- TRAJEX **不做聚类、不分配行人 ID**，它只做"逐轨迹特征化 + 角色 1-NN 判定"；
- TRAJEX 同时声明 _自己写的特征表_ 与 _下游计算层写的结果表_ 的 DDL，但只对前者写，后者只读——DDL 作为契约，权责由代码保障。

## 2. 数据来源与去向

### 2.1 入口（多源切换）

感知层的入口设计为**双轨可切换**——既支持流式增量，又支持批式回放：

- **流式入口**：消息队列订阅，消费实时上传的轨迹消息；
- **批式入口**：从数据仓库按时间窗 + 租户拉取历史轨迹。

每条轨迹消息包含轨迹 ID、摄像头标识、店铺/公司标识、时间戳序列、坐标序列、图像 URL 列表（图像本身存在对象存储里，按需下载）。

### 2.2 出口（特征表）

四张按天分区的宽表，分两个场景（出入口 / 店内）× 两种粒度（图级 / 轨迹级）：

- **图像级特征表**：每条轨迹的多张代表帧各算一行——存上装检测 bbox、上装特征向量、角色/工服标签 + 分数，以及可选 / 预留的质量字段。
- **轨迹级特征表**：每条轨迹聚合一行——存轨迹起止时间、坐标统计、行人特征向量（出入口/店内独立特征 + 跨场景统一特征）、角色融合分数。

每行都打上**模型版本标签**，形成"哪个模型在哪一天产出了这个向量"的完整血缘。

## 3. 模型推理：三类模型 + 推理服务统一暴露

感知层的核心算力消耗在三类模型上：

1. **服装/身体检测**：定位上装、头部、安全帽等 bbox + label，是后续所有特征/角色判定的前置；
2. **上装特征**：把检测到的上装裁剪后映射到固定维度向量，喂给底库查询；
3. **行人 ReID 特征**：把整条轨迹的若干代表帧编码为单一向量；同时支持旧主特征轴与新统一特征轴两条独立特征轴并行（影子部署细节见后文第 7 节）。

可选 / 预留第四类：**质量分类器**——给每张图打一个"是否值得用"分数，便于下游特征表内做加权或丢弃；当前表结构保留质量版本与结果位，实际推理链路可按配置启停。

### 3.1 推理服务部署模式

所有模型走**统一推理服务**，关键工程优化：

- **本机共享内存直连**：与推理服务同机部署时，绕过 HTTP 序列化，用系统共享内存传输输入张量——是 GPU 节点上 ReID 这种"小批量但高频"场景的关键加速点；
- **异步并发**：检测、上装特征、ReID 等独立模型同时发起推理请求，让 GPU 流水线打满；
- **多进程 worker + 队列协调**：感知层的吞吐瓶颈通常是图片下载与前处理而不是 GPU 推理，靠多进程横向扩展；
- **批量与序列采样**：轨迹图像按固定长度上限等距截取，再按序列长度重排成小批次推理，最后平均池化得到轨迹级表征——避免逐帧推理浪费 GPU。

## 4. 库查询作为特征：1-NN 检索内联到特征表

感知层做了一件**容易被忽视的关键决策**：把"角色判定"（顾客 / 店员 / 外卖 / 快递 / 保安）以 **1-NN 库查询** 的方式融进特征表，而**不是**留给下游做。

每条轨迹完成特征提取后，**同步**查询若干底库：

- **通用角色服饰库**（跨公司共享的外卖/快递/安保等角色样本）→ 命中即直接判该角色；
- **公司 / 店铺工服库**（按组织隔离的员工制服样本）→ 命中即可作为店员候选；
- **店员 ReID 底库**（真实店员历史轨迹特征）→ 命中即作为店员置信加权。

多路查询并发执行，结果直接写入特征表中的角色类型与置信度字段。这个设计把"角色判定"从离线聚类管线里上移到了感知层，让下游每个簇都自带角色证据。

## 5. 去重：感知层无幂等假设

边缘上传 + 消息队列 + 多 worker 消费的组合天然带**重复处理风险**：

- 上游消息队列是 at-least-once 语义；
- 历史回放（rerun）会把已处理过的轨迹重新发一遍；
- 多 worker 偶发会重复消费同一消息。

感知层用**两层精确集合去重**，按天滚动过期：

- **生产端去重**（可选）：在写入消息队列之前，批量检查已处理集合，已处理则跳过；
- **消费端去重**（必选）：worker 在下载图片之前再查一次集合，命中则丢弃整条消息。

历史经验：**近似去重结构因为假阳性会丢失新轨迹，最终被替换为精确集合**——精度优先于内存。回放上下文会显式绕过去重网关，这样人工触发的"完整重算"不会被去重误杀。

## 6. 三种回放模式

模型升级、bad case 修复、Redis 去重态丢失等场景需要批量回放。TRAJEX 提供三种独立工具：

| 模式                  | 输入             | 是否过去重   | 是否重算特征       | 典型用途                    |
| --------------------- | ---------------- | ------------ | ------------------ | --------------------------- |
| **Scheduled rebuild** | 配置驱动定时任务 | 受去重约束   | 是                 | 日常补漏（小时/天周期）     |
| **Manual rerun**      | 人工指定时间窗   | **绕过**去重 | 是                 | bad case 排查、模型 cutover |
| **Cache rebuild**     | 时间窗           | —            | 否，仅重建去重集合 | 灾备恢复、去重态丢失        |

每行特征都带**来源血缘字段**（流式 / 批式 / 回放 / 重建 / ...），下游分析时可按来源做漏斗图、问题定位。

## 7. 模型升级：影子特征轴并行

ReID 的特征模型每年都在演进（详见 `ReID-Embedding-Models.md` 的技术谱系）。感知层用 **影子部署模式** 让新旧模型共存：

- 特征表预留两套逻辑特征轴：旧主特征轴 + 新统一特征轴；每套都包含向量列与模型版本列；
- 两个模型可以同时跑，两个向量列同时写；
- 下游 HIDALGO 计算层可以选择消费哪一套（或同时跑双轨实验）；
- cutover 是配置开关，不是数据迁移；
- 回滚也是配置开关，秒级生效。

## 8. 库查询的 hybrid 存储

感知层底层向量存储正在从**专用向量库**逐步迁到**关系库的向量扩展**：

- 每个查询点暴露后端切换开关，按集合、按租户灰度迁移；
- 已迁完的：通用角色服饰库、公司 / 店铺工服库；
- 仍在专用向量库的：店员 ReID 底库（数据规模大，迁移成本高，留待第二阶段）；
- **关键设计**：dispatcher 在处理逻辑入口处分支，迁移过程不动业务代码——只动配置。

这是把"大规模存储迁移"转化为"按集合灰度的渐进切换"的标准范式。

## 9. 运维面：管理后台 / API / 告警三件套

感知层的 day-2 运维独立于核心计算，分三个子服务：

- **管理后台**：浏览历史轨迹、可视化轨迹 + 图片、人工标注店员（把某条轨迹的 ReID 特征推入店员库）；
- **HTTP API**：暴露给上游业务的轻量 API，包括"添加店员特征"和"分页查询轨迹"；
- **告警服务**：定时扫描"未匹配设备"集合，按组织分发告警——填补"边缘相机硬件标识 → 业务公司/店铺映射缺失"这种排查盲区。

### 9.1 关键 metrics

按指标采集体系暴露系统与业务指标：

- 系统层：内存、CPU、线程、文件描述符；
- 业务层：轨迹处理总数、处理耗时分布、队列深度、图片下载耗时；
- 后端层：每个外部依赖（缓存 / 关系库 / 向量库 / 数据仓库）的延迟与错误计数；
- 模型层：`{model, operation}` 二元标签的推理延迟。

**关键技巧**：回放模式的指标用独立命名空间，避免回放任务污染日常 SLI dashboard。

## 10. 设计哲学小结

- **感知与计算分离**：感知层不做 ID 分配，把"行人是谁"这种全店级判断留给下游；
- **库查询 ≠ 聚类**：把"角色判定"（1-NN）和"行人聚合"（多步聚类）做语义分层，前者烤进特征表、后者留给离线批；
- **DDL 即契约**：生产者/消费者权责通过代码而非数据库权限隔离，DDL 是双方共同维护的接口；
- **特征行带血缘**：模型版本与来源字段让历史数据可追溯、可重算；
- **影子特征轴并行**：模型升级 = 多写一组向量 + 版本列，cutover = 改配置，不动数据；
- **配置驱动渐进迁移**：底层存储切换不进代码，按集合 / 租户 / 时间窗灰度。
