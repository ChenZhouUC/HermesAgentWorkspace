# ReID 系统架构与重构方案

本文记录线下门店行人重识别（ReID）系统的端到端架构思路、聚类策略与基础设施选型原则。
ReID 是上传轨迹/图像信息的第一道处理；其输出（行人 ID、行人角色）会被下游的客流后处理进一步加工成业务指标，后处理逻辑见 `Customer-Flow-Post-Processing.md`。

**命名边界**：HIDALGO 是 `project_hidalgo_reid` 的顶层项目名称，也常用于指代其中负责后续聚合、角色推断与结果回写的 ReID 计算层服务。TRAJEX 是同一项目下负责特征推理的子项目/服务。本文说"ReID 核心"或"本系统"时，默认指 HIDALGO 计算层，不包含 TRAJEX 感知层。

## 1. 系统定位与重构动因

线下零售门店在出入口与店内部署多路摄像头，**边缘计算盒子（运行 Edge ALGO）**实时处理视频流，提取出入口/店内事件，并向云端上传**带时空坐标的人体轨迹切片、抓拍图像及年龄性别等属性**。ReID 系统的职责是：

1. 在同一门店内把不同摄像头、不同时间窗的轨迹聚合为同一行人；
2. 识别每条轨迹的角色（顾客 / 店员 / 外卖 / 快递 / 保安）；
3. 把识别结果输出到下游做客流统计、接待识别、人货匹配。

老系统的典型痛点：

- 算法只用 Embedding + 单阶段聚类，全量门店上限受限、门店间方差极大；
- 工程层是单点架构，图片下载与向量计算成为瓶颈；
- 用专用向量数据库单独存储 Embedding，与门店组织架构（公司 / 门店 / 摄像头）的层级映射不自然，且与业务库存在双写一致性问题。

重构方向：聚合质量提升至生产可用水平、改造为按店可扩展的批处理架构、资源利用率上调、故障自愈。

## 2. 分层职责划分

**重要的边界**：HIDALGO 计算层核心定位为**离线批处理引擎**——它不负责实时采集、模型推理、调度编排。这些职责分属上下游：

- **极前置（Edge ALGO 边缘算法）**：运行在门店算力盒子（如 RK3576, Sophon 等）上，处理原生视频流，提取轨迹事件、抓拍图像、时间戳序列与基础属性，并内置弱网缓存重传机制。
- **上游（TRAJEX 感知层）**：接收边缘端通过消息队列上传的数据，由模型推理服务负责把图像→检测→特征向量；把上装类别、角色得分、行人特征预先算好后写入特征表；上游会查询全公司共享的工服底库与按公司隔离的特殊角色/店员底库，把命中结果作为列直接落盘到特征表。
- **核心（HIDALGO 计算层）**：从特征表读已计算好的 Embedding 与轨迹，做角色推断 / 重判 + 多层级聚合 + 结果回写。**当前主流程不调用模型、不写消息队列**，整个过程是函数化的 `(店铺, 起止时间, 模式) → 结果`。
- **下游（编排层）**：DAG 调度器按店/按时间窗周期性触发核心，并维护任务级失败回滚、补数、灰度。本系统不内嵌调度逻辑，只暴露一个轻量级的任务进度表供调度层做幂等与续跑。

这种**"采集 / 计算 / 编排"三段切分**是后续所有可演进性的基础：模型升级不动批处理；调度策略变化不动算法；算法改阈值不动调度。

## 3. 三模式对偶：Entrance / Interior / Coupling

核心运行有三种模式，**每次调用只跑一种**：

- **Entrance**：出入口轨迹去重——把同一顾客多次出入口轨迹聚为同一行人；
- **Interior**：店内轨迹组装——把店内多摄像头碎片轨迹聚为完整店内动线；
- **Coupling**（生产主模式）：先用 Entrance 模式产出"锚轨迹"，再把店内轨迹匹配挂载到对应行人 ID 上，形成"全店动线"。

Coupling 是 **Entrance 复用 + Interior 匹配** 的组合，不是独立算法路径；这种设计避免了"全店"这条复杂主线被特化逻辑污染。

## 4. 多层级聚合策略

**核心方法学**：聚合从来不是"一次跑一个聚类算法"，而是把"由严到松、由同向到异向、由轨迹到批次"的判断切成多个独立 tier，每 tier 独立配参、独立通过/失败。底层数学统一是 **相似度图上的连通分量** 计算——不是经典 HAC 的 single linkage，而是按"动作方向掩码 + 摄像头一致性掩码 + 时间窗掩码"裁剪相似度图后求传递闭包。每个 tier 控制 _mask 范围_ + _相似度阈值_ + _合并模式（all / max / exclusive）_，把"匹配标准"几何化为图论问题。

实际生产层级（精简到方法学层面）：

### 出入口聚合的多个 tier

按"先确定再放宽"的顺序：

1. **同方向单体**：先在确信的、单方向的孤立轨迹之间做去重；
2. **同方向聚合**：把同方向多帧轨迹合并；
3. **跨方向中期合并**：跨方向但时间相近的轨迹做中等阈值合并；
4. **异方向单体**：处理一进一出的成对轨迹（同一顾客）；
5. **异方向覆盖合并**：进出时间窗重叠的成对轨迹；
6. **异方向适配合并**：进出特征贴合的剩余成对轨迹；
7. **总结兜底**：宽松阈值扫尾剩余碎片。

每个 tier 内部还有"多轮阈值递减"机制——相同的判定条件下，阈值会从高到低衰减若干轮，让"确信度高的"先 lock-in，"边缘的"后逐步收敛。

### 店内轨迹组装的多档

店内组装按相似度档位（高 / 中高 / 中 / 中低 / 低）依次推进；同一组装目标可在不同档位接收新成员，直到无新增。

### 跨场景匹配的多阶段

把店内轨迹挂载到出入口行人锚点时，分为：

1. **锚定（Anchor）**：用最确信的店内轨迹绑定到一个出入口锚；
2. **匹配（Match）**：剩余轨迹按相似度逐个挂载；
3. **补充（Supplement）**：用次级信号（时长、共现摄像头）再扫一轮；
4. **补偿（Compensate）**：兜底召回，允许低阈值。

## 5. 角色判定的两阶段

角色判定**不是单一前置步骤**，而是穿插在聚类前后：

### 5.1 预聚类（per-image / per-trajectory）

- **逐图打分**：根据上装检测的几何信号（头盔标签触发外卖/快递、徽章标签触发店员）+ 上装相似度（命中工服库判店员、命中特殊角色库判该角色）逐图给出 `top_biz_type`；
- **逐轨融合**：把全轨迹的图级别打分加权聚合到轨迹 `ped_type`，店员判定用**店内特征模型分** 与 **上装统一模型分** 的加权融合，避免单一信号在光照/角度上的不稳定；
- **预筛**：聚类前用 `ped_type` 白名单筛掉与该模式无关的角色（如店内组装阶段直接剔除店员，避免店员轨迹污染顾客聚类）。

### 5.2 聚类后（per-cluster）

- **重判**：聚类完成后，对每个簇取众数 `ped_type`；
- **徽章比例覆写**：若簇内徽章命中轨迹占比足够高，强制改判为店员；
- **融合分兜底**：若簇内某轨迹的"店内特征 × 工服特征"融合分超阈值，再次改判为店员。

这种**"前判定 + 后重判"** 设计承认了"聚类本身可能修正角色"——一条孤立的高相似度店员轨迹会被识为顾客，但当它和真店员的其他轨迹合到同一簇后，整簇会被改判回店员。

## 6. 基础设施选型

### 6.1 向量列与业务列同表

老系统的常见做法是用专用向量数据库 + 业务关系库双写，存在一致性与权限治理问题。重构思路：把向量列直接放进关系库（利用其向量扩展），让向量与业务字段（公司 / 门店 / 轨迹 ID / 时间戳）落在同一张表里。

```
Database  ─▶ Schema    ─▶ Table      ─▶ Partition ─▶ Column
功能区        公司          特征/结果     时间天分区     向量 / 业务字段
```

按店扩缩、按天归档、按公司隔权限全部走一套 ACL 与 DDL；向量列与业务列同事务读写，从源头消除"向量库与业务库双写一致性"问题。

### 6.2 推理服务的本机优化

TRAJEX 感知层的模型推理通过推理服务统一暴露；与该服务同机部署时，走**共享内存直连**而不是 HTTP——绕过反序列化与 socket 开销。这是把端侧重的推理服务挪到生产环境时的关键加速点。

### 6.3 外部调度器 + 本地进度表

生产环境用外部 DAG 调度器周期触发本系统，本系统**只暴露一张轻量任务进度表**记录每次调用的：店铺 / 时间窗 / 模式 / 状态 / 耗时。这样：

- 算法侧改逻辑不动调度器；
- 调度器 retry / backfill / 灰度策略不动算法；
- 进度表本身可作为补数与幂等的真理源（同样的 (店铺，时间窗，模式) 重跑会被跳过）。

### 6.4 多套环境配置中心

把数据库连接 / 模型路径 / 桶名等环境差异封装到一个小型配置中心（开发 / 预发 / 生产 三表），用环境变量切档，代码里只引用逻辑名而不写具体地址。

## 7. 反作弊与采样策略

- **按日采样上限**：单店单日的轨迹数有上限，超出后做**有偏下采样**——优先保留"进店"事件而下采样"出店"事件，因为业务侧客流统计的关键指标是进店人数；
- **配置可按店覆写**：上限本身按店可调，特殊大店可调高，避免一刀切伤精度。

## 8. 评估指标分层

为了解耦**特征模型** 与 **聚类策略** 与 **下游业务**，评估指标按层：

- **角色标签层**：精确率 / 召回率（按"店员 / 顾客 / 三类特殊角色"分别评）；
- **聚类策略层**：双向熵——`-Σ p·log(p)`，一边算"过度聚合"（同一聚类对应多个 GT 行人 → 熵高），一边算"过度切分"（同一 GT 行人散落多个聚类 → 熵高）。两边都要小才算稳。

更上层的特征模型性能（Rank-N / mAP / mCP）由**训练侧**独立评估，不在本系统职责内——这是上下游分层职责的体现。

## 9. 配置覆写与每店调优

每店运营特征不同（光照、客流密度、出入口几何），系统在默认参数之外提供一张**每店参数覆写表**：以 JSON 列存放每个店铺的差异化参数，运行时与默认参数做 deep merge。

这种设计的关键价值：**新策略上线只是改这张表里某些店的配置项**，不动主代码、不动镜像、不动调度器；按店灰度成为一次 SQL 操作，回滚也是一次 SQL 操作。

## 10. 历史变体与实验代码并存

仓库中并存两类辅助代码：

- **历史变体**：旧版核心引擎留在另一套同名 `_dt` 后缀的模块中，做对照与 A/B；
- **探索代码**：包含若干实验性聚类策略，不在生产路径上调用。

这部分**不应进入主体方法学**——它们的价值在于回滚后备与新策略实验场，而不是描述系统现状。
