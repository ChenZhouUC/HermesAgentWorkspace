# TRAJEX vs HIDALGO (ReID 感知层 vs 计算层的分工对比)

线下零售门店 ReID 系统被切分为两段独立服务：[[trajex|TRAJEX]]（感知层，图像 → 特征 + 角色）和 [[hidalgo|HIDALGO]]（计算层，特征 → 行人 ID）。本页对比这两段的职责边界、技术取舍与适用场景，提炼出"何时该把 ReID 拆成两段、何时应该合并"的决策依据。

## 分工对比表

| 维度         | TRAJEX（感知层）                       | HIDALGO（计算层）                               |
| ------------ | -------------------------------------- | ----------------------------------------------- |
| 输入         | 由 [[edge-algo                         | Edge ALGO]] 抽取的轨迹消息 + 脱敏图像地址       | TRAJEX 写入的特征表行 |
| 输出         | 特征表（图像级 + 轨迹级）              | ReID 结果表 + 业务宽表                          |
| 工作负载     | GPU 推理（检测、特征模型）             | CPU 计算（连通分量、规则与阈值聚合）            |
| 处理粒度     | 单条轨迹                               | 全店级、跨摄像头、跨时间窗                      |
| 触发模式     | 流式消费 + 批回放                      | 编排器周期触发（按店按时间窗）                  |
| 延迟特征     | 单条轨迹端到端秒级以内                 | 全店一次跑数分钟到小时                          |
| 状态依赖     | 去重集合（按天滚动）                   | 任务进度表（按店 × 时间窗 × 模式）              |
| 模型用法     | 推理（用模型生成数据）                 | 不调模型（消费已生成的特征）                    |
| 算法核心     | 1-NN 库查询（见 reid-library-lookup）  | 多层级连通分量聚合（见 multi-stage-clustering） |
| 角色判定     | 逐轨迹打"角色证据"                     | 簇内多数表决 + 徽章比例覆写                     |
| 失败重试     | 消息级重试 + 去重幂等                  | 任务级 backfill + 进度表幂等                    |
| 与外部接口   | 消息队列消费 + 数据仓库拉取 + 推理服务 | 只读 / 写关系库；不出库                         |
| 主要运维风险 | 模型推理超时、GPU OOM、底库版本错乱    | 阈值飘移、bad case 积累、上游 DDL 延迟          |

## 核心 Trade-offs

- **GPU 工时利用 vs CPU 计算成本**：感知层把所有 GPU 推理压在一段，让 GPU 资源池化、最大化吞吐；计算层完全 CPU，可水平扩在普通节点上——一台 GPU 训出来的特征喂给十台 CPU 跑聚类是常见配比。如果合在一起跑，要么 GPU 闲置等聚类、要么聚类阻塞特征产出。
- **流式 vs 批式两套触发节奏**：感知层是事件驱动（消息到达即处理），延迟敏感；计算层是时间窗驱动（按店按小时按天），关心吞吐而非单次延迟。两套节奏混用一个调度器会让 SLO 难定义。
- **模型升级冲击面**：感知层模型升级走 [[model-shadow-deployment|配对特征轴影子部署]]，计算层完全无感；如果合在一起，模型升级需要同时回滚聚类策略，blast radius 翻倍。
- **去重模型差异**：感知层用按天滚动的精确集合去重（流式 + 高频）；计算层用任务进度表按 `(店铺, 时间窗, 模式)` 三元组去重（批式 + 低频）。两套去重机制都成立但若合并会被迫选边。
- **代码可观测性**：拆两段意味着 schema 接口 + 指标 + 看板各成体系，可定位"是特征出问题还是聚类出问题"；合并后这种归因变得困难。
- **代价**：拆两段需要维护 [[schema-as-handoff-contract|DDL 契约]]、跨服务部署、双倍运维面（管理后台 / API / 告警）；小规模场景这些成本可能不值得。

## When to use which split

**应该拆成 TRAJEX + HIDALGO 计算层两段**：

- **GPU 资源稀缺、需要池化**：特征提取是 GPU bound，聚类是 CPU bound，混在一起会让 GPU 等聚类；
- **模型迭代节奏远快于聚合策略迭代**：把"模型升级"和"业务逻辑升级"沿管线切开，让两边独立演进；
- **流式 + 批式混合负载**：实时上传需要尽快出特征，但客流统计可以等到闭店后批跑——两种节奏的 SLO 不同；
- **多店扩张场景**：感知层按店水平扩，计算层按公司池化，两边各自满足扩展模型；
- **需要在感知层就拿到角色证据**：见 [[reid-library-lookup]] —— 让聚类用角色作为前置过滤而非事后判定。

**应该合并为单服务**：

- **场景规模小**：只有少量店铺，每天轨迹量不大，跑 GPU 闲置时间长，没必要拆两段；
- **聚类极简（如纯阈值匹配，无多 tier）**：算法本身轻量，与特征提取放一起也不会成为瓶颈；
- **运维团队规模小**：维护两套服务、两套监控、两套灰度的工程成本可能超过收益；
- **延迟严苛到只能在线**：若业务要求"上传即得 ID"，完全没法走批，那拆开反而引入额外 hop。

## 与其他系统的可类比性

这种"前后端分层 + DDL 契约 + 影子部署"的组合不限于 ReID，可推广到：

- 推荐系统：感知层做用户/物品特征提取，计算层做匹配/排序——同样适合拆；
- 实时风控：感知层做特征提取（设备指纹、行为序列），计算层做规则引擎/图聚类——拆与否取决于实时 SLO；
- 广告检索：感知层做 embedding 召回，计算层做精排——和 ReID 同构。

把"何时拆"这个问题工程化成"GPU/CPU 工作负载形态 + 时序/批式 SLO 差异 + 模型/策略迭代节奏差异"三维评分，几乎在任何 ML 数据管线选型中都能复用。
