# Agentic Demo Pipeline (双模型自动化研发流 v2.0)

当用户提出一个产品 Demo 或功能实施需求时，调用此 Skill。为了保证执行透明度和高质量，本流水线**必须分阶段执行并向用户阶段性汇报**。

## 🎯 核心工作流

### 阶段 1：深度需求调研与规划 (回合 1)

1. **初始化**：根据需求提取英文项目名，在 `~/.hermes/workspace/<project_name>` 创建项目专属目录。
2. **委派 Codex 调研**：使用 `delegate_task` 工具唤起 `codex`。
   - **goal**: "深度调研用户的需求并生成产品与技术规划。你必须极其详实地调研：至少进行 5 次以上的 web_search，浏览并提取 20 个以上的相关网页内容。完成后将规划输出到 ~/.hermes/workspace/<project_name>/PLAN.md。"
   - **context**: "用户需求：[插入需求]。在 PLAN.md 中，不仅要包含详实的业务与竞品分析，还要包含架构设计说明，为后续扩展到 App 或其他端做好准备。"
3. **阶段性汇报**：Codex 执行完毕后，不要立刻执行后续操作。**你必须在此次回复中停下来，向用户输出一份阶段性汇报**：告知当前进度、调研了多少个网页/竞品、PLAN.md 的核心结论，并询问用户：“前期规划已完成，是否立刻开始实施代码？”

### 阶段 2：模块化代码实施与文档生成 (回合 2)

1. **委派 Copilot 实施**：在用户确认后，使用 `delegate_task` 唤起 `copilot`，指定 `acp_args: ["--model", "claude-sonnet-4.6"]`。
   - **goal**: "根据 PLAN.md 编写 Demo 代码，并生成配套 README.md。代码需要轻量，但必须兼顾极强的兼容性与可扩展性（如采用数据与视图分离、清晰的组件化注释，甚至剥离出独立的 JS 逻辑层），以便未来无缝迁移到移动端 App 或 React/Vue 框架。"
   - **context**:
     - "产出 1：在工作目录下生成 Demo 代码文件（必须高保真且易于扩展）。"
     - "产出 2：在工作目录下必须生成 README.md，简要介绍代码的文件结构、组件化思路、是否需要环境依赖以及如何运行测试。"
2. **最终验收与汇报**：Copilot 执行完毕后，向用户汇报最终生成的文件列表、`README.md` 的核心说明，以及如何运行该 Demo。

## 🚨 核心准则

1. **深度优先**：调研环节必须动用充足的搜索额度，切忌走马观花。
2. **架构可扩展**：Demo 绝对不能是一团“意大利面代码”。HTML 里的 DOM 结构、样式与状态必须高度解耦。
3. **及时沟通，切忌自嗨**：严禁在一次响应中把调研和代码一口气全跑完。必须严格遵循“规划 -> 汇报 -> 等用户确认 -> 实施 -> 汇报”的节奏。

## ⚠️ 常见踩坑 (Pitfalls)

- **子 Agent 产出超时 (Timeout)**：通过 `delegate_task` 唤起 Copilot 强写几百上千行的静态 Tailwind DOM 时，极易触发模型超时（如 10+ 分钟未完成）。**解决办法**：在 Prompt 中强制要求采用“数据与视图解耦”的架构（如 Vue3 `v-for` + 本地 JSON mock）。这不仅符合工程最佳实践，还能大幅压缩 HTML DOM 代码量，完美规避超时问题。
